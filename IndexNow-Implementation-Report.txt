1. SUMMARY (non-technical)
------------------------------------------------------------------------
We added an automatic "IndexNow" integration to the website. IndexNow is a
protocol supported by Bing (and Yandex and other engines) that lets a website
notify the search engine THE MOMENT a page changes, instead of waiting days or
weeks for the search engine to re-crawl on its own.

In practice: when we edit and deploy a service page, Bing is now notified within
seconds and re-crawls that page much faster. This shortens the time between
publishing an SEO/content change and that change being reflected in Bing search.

Note: IndexNow is supported by Bing/Yandex, NOT Google. Google is unaffected and
continues to crawl normally.


------------------------------------------------------------------------
2. BUSINESS VALUE
------------------------------------------------------------------------
- Faster Bing pickup of content/SEO edits (minutes/hours vs. days/weeks).
- Fully automatic going forward — no manual submission step per page.
- Targets the service pages, which are the site's primary SEO landing pages.
- Complements the recent service-page SEO/meta-tag overhaul by getting those
  updates in front of Bing quickly.

4. HOW IT WORKS
------------------------------------------------------------------------
Change detection:
  - Scans the service-page content folders (Views/ServicePage/Content).
  - Reads each page's canonical URL directly from its articlemetas file.
  - Uses the newest file-modified time of the page's article + metas files as the
    "last changed" signal (the same signal the site already uses for the page's
    article:modified_time meta tag).
  - Compares against a saved snapshot; only NEW or CHANGED pages are submitted.

Triggers (automatic):
  - On startup: ~20 seconds after every deploy / app restart.
  - Periodic: a repeating re-scan (configurable; currently set to every 72 hours)
    as a safety net for changes made without a restart.

First-run safety:
  - The very first run records a silent baseline and submits nothing, so a fresh
    deploy does not flood Bing with the entire site at once. Only subsequent
    changes are submitted.

Manual triggers (on demand, password-protected):
  - /IndexNow/Status  — read-only health/config check (no submission).
  - /IndexNow/Scan    — force a scan now (optional "force" to submit on first run).
  - /IndexNow/Ping    — submit one or more specific URLs immediately.

7. HOW TO VERIFY AFTER DEPLOY
------------------------------------------------------------------------
  1. Open the key-file URL above -> it should display the key (not a 404).
  2. Open /IndexNow/Status?key=<TriggerSecret> -> should report configured = true
     and a tracked URL count (~98).
  3. Submit one page via /IndexNow/Ping (or Bing's "Request indexing" button) and
     confirm it in Bing Webmaster Tools -> URL Inspection (the "Last crawled" date
     should update within hours).
  4. To push the existing batch of SEO edits at once, run /IndexNow/Scan with the
     "force" option a single time after the first deploy.

8. LIMITATIONS / NOTES
------------------------------------------------------------------------
  - Scope: automatically covers SERVICE pages. Other page types (blog, locations,
    etc.) can be added to the configuration later, or pushed manually via the Ping
    endpoint. (Easily extended.)
  - IndexNow notifies; it does not guarantee instant indexing. Bing re-crawls on
    its own schedule (typically minutes to hours after the notification).
  - Bing/Yandex only — Google does not use IndexNow.
  - Two items in the commit are environment/secret-sensitive and worth a follow-up
    review: (a) the manual-endpoint TriggerSecret currently lives in
    appsettings.json (consider moving secrets to server environment variables);
    (b) App_Data/indexnow-state.json is runtime state that was committed — it will
    self-correct on first run and can optionally be git-ignored.
  - Separate, related finding (not part of this commit): the site's sitemap.xml
    <lastmod> dates are all frozen at a single date, which weakens the freshness
    signal for Google. Recommended as a follow-up so Google also picks up changes
    faster (IndexNow does not help Google).
